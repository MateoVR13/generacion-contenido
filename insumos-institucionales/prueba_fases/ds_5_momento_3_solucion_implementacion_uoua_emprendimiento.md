## Contenido desarrollado

### 1. Proposito del Momento 3

El Momento 3 convierte el aprendizaje del reto y la ideacion en una representacion concreta de solucion. En los momentos anteriores el estudiante definio un problema real, caracterizo actores, formulo hipotesis, priorizo incertidumbres y preparo instrumentos exploratorios. Ahora debe decidir que solucion inicial merece ser representada y como implementarla de manera controlada para seguir aprendiendo.

En esta asignatura, solucion e implementacion no significan lanzar una empresa, programar un producto final, fabricar un prototipo funcional costoso o prometer impacto. Significan construir una version de baja fidelidad que permita visualizar, explicar, simular o probar una propuesta de valor inicial con actores pertinentes.

Este momento produce cuatro evidencias centrales del portafolio Gate 1:

1. Matriz evidencia-solucion.
2. Propuesta de valor inicial.
3. Prototipo de baja fidelidad.
4. Bitacora de decisiones de diseno e implementacion.

Estas evidencias preparan el Momento 4, Validacion y evaluacion. Si el prototipo no se conecta con el problema, la validacion medira preferencias superficiales. Si se conecta con hipotesis y evidencia, la validacion podra aportar aprendizaje util para decidir continuidad, pivote, reformulacion o cierre.

### 2. Solucion inicial: respuesta para aprender, no producto final

Una solucion inicial es una respuesta preliminar al problema validable. Su valor no esta en estar terminada, sino en permitir aprendizaje. Debe estar relacionada con el actor, el contexto y las hipotesis priorizadas. Tambien debe poder representarse con recursos simples y dentro del tiempo del curso.

El estudiante debe evitar confundir solucion inicial con producto final. Una aplicacion completa, una plataforma comercial, un dispositivo funcional, una campana institucional o una empresa formal suelen estar por fuera del alcance de esta asignatura. En cambio, un flujo simulado, un boceto, una maqueta, un storyboard, una pantalla navegable basica, un guion de servicio o una plantilla de proceso pueden ser suficientes para validar si la direccion de la propuesta tiene sentido.

| Concepto | Que significa en el curso | Que no significa |
|---|---|---|
| Solucion inicial | Alternativa preliminar conectada con el problema y las hipotesis. | Producto final listo para mercado o implementacion institucional. |
| Implementacion controlada | Representar o simular la solucion en un entorno limitado y etico. | Lanzamiento publico, venta real o intervencion sin autorizacion. |
| Prototipo de baja fidelidad | Representacion simple para aprender de actores y decisiones. | Desarrollo tecnico completo, version definitiva o promesa de resultado. |
| Propuesta de valor inicial | Explicacion de como la solucion podria aliviar dolores o crear ganancias. | Garantia de impacto, modelo de negocio completo o prueba de mercado. |

Ries (2011) plantea que construir debe estar al servicio de medir y aprender. Brown (2009) resalta el prototipado como forma de pensar con objetos, imagenes, historias y experiencias. Osterwalder et al. (2014) permiten relacionar la solucion con trabajos, dolores y ganancias. En conjunto, estas ideas ayudan a que el estudiante construya poco, pero aprenda mucho.

### 3. De la evidencia a criterios de diseno

Antes de prototipar, el equipo debe traducir su evidencia en criterios de diseno. Un criterio de diseno es una condicion que la solucion debe cumplir para responder al problema. No nace de gustos del equipo, sino de lo aprendido sobre actores, contexto, restricciones y valor.

Por ejemplo, si la evidencia muestra que los usuarios tienen poco tiempo para decidir donde depositar residuos, un criterio de diseno puede ser "la senalizacion debe comprenderse en menos de diez segundos". Si una microempresa tiene baja alfabetizacion digital o poco tiempo, un criterio puede ser "el registro debe poder completarse en una accion corta y sin duplicar trabajo". Si un laboratorio requiere seguridad y permisos, un criterio puede ser "la solucion no debe publicar informacion sensible sobre equipos o horarios restringidos".

| Fuente de evidencia | Pregunta para convertirla en criterio | Criterio de diseno posible |
|---|---|---|
| Entrevistas | Que necesidad, dolor o restriccion se repite? | La solucion debe reducir confusion en el primer contacto del usuario. |
| Observacion | Que comportamiento o friccion aparece en contexto? | La solucion debe ubicarse en el punto donde ocurre la decision. |
| Encuesta breve | Que patron preliminar aparece entre actores? | La solucion debe priorizar el factor reconocido como mas critico. |
| Mapa de actores | Que actor decide, usa, bloquea o facilita? | La solucion debe ser comprensible para usuario y decisor. |
| Restricciones eticas | Que informacion no debe recolectarse o exponerse? | La solucion debe funcionar sin datos sensibles. |

La matriz evidencia-solucion permite ordenar este paso:

| Evidencia recolectada | Interpretacion prudente | Criterio de diseno | Implicacion para el prototipo |
|---|---|---|---|
|  |  |  |  |
|  |  |  |  |
|  |  |  |  |

La palabra "prudente" es clave. En validacion temprana no se deben sacar conclusiones definitivas desde pocas respuestas. La evidencia inicial orienta, no sentencia.

### 4. Propuesta de valor inicial

La propuesta de valor inicial explica por que una solucion podria ser relevante para un actor. No es un eslogan ni una promesa comercial. Es una formulacion breve que conecta actor, problema, dolor o ganancia esperada y alternativa de solucion.

Osterwalder et al. (2014) proponen pensar en trabajos, dolores y ganancias. Para el nivel de esta asignatura, el estudiante puede usar esa logica de manera sencilla:

| Elemento | Pregunta guia | Respuesta esperada |
|---|---|---|
| Actor | Para quien se propone la solucion? | Usuario, cliente, empresa, comunidad o decisor definido. |
| Trabajo o necesidad | Que intenta lograr ese actor? | Actividad, proceso, decision o resultado esperado. |
| Dolor | Que dificultad, costo, perdida o frustracion enfrenta? | Evidencia conectada con el problema. |
| Ganancia esperada | Que mejora seria valiosa? | Ahorro de tiempo, claridad, reduccion de error, confianza, sostenibilidad. |
| Solucion inicial | Que alternativa se prototipara? | Boceto, flujo, servicio, tablero, senal, formato o simulacion. |
| Diferencia relevante | Que hace que esta alternativa sea pertinente para el contexto? | Criterio de diseno derivado de evidencia. |

Estructura sugerida:

```text
Para [actor] que necesita [trabajo o necesidad] y enfrenta [dolor evidenciado], proponemos [solucion inicial] que podria [ganancia esperada], porque [criterio de diseno o evidencia que la sustenta].
```

Ejemplo:

```text
Para estudiantes que requieren equipos en un laboratorio compartido y enfrentan esperas por falta de informacion sobre disponibilidad y reglas, proponemos un tablero visual de consulta y reserva simulada que podria reducir reprocesos, porque la evidencia exploratoria muestra confusion en los pasos de solicitud.
```

Esta formulacion debe considerarse provisional. En el Momento 4 podra confirmarse, ajustarse o descartarse.

### 5. Seleccion de alternativa de solucion

En DS-4 el equipo genero alternativas. En DS-5 debe elegir una para prototipar. La seleccion no debe depender solo de preferencia estetica, novedad tecnologica o facilidad del equipo. Debe justificarse con criterios academicos y de validacion.

| Criterio de seleccion | Pregunta guia | Senal de calidad |
|---|---|---|
| Coherencia con problema | La alternativa responde al problema definido? | Se puede trazar desde reto, evidencia e hipotesis. |
| Relacion con actor | El actor podria entenderla, usarla o evaluarla? | El lenguaje, formato y contexto son pertinentes. |
| Viabilidad de baja fidelidad | Puede representarse sin construir producto final? | Se puede hacer con boceto, plantilla, maqueta, storyboard o simulacion. |
| Aprendizaje esperado | Permite validar una hipotesis prioritaria? | Genera preguntas claras para el Momento 4. |
| Cuidado etico | Puede probarse sin exponer datos o afectar personas? | No requiere informacion sensible ni intervenciones no autorizadas. |
| Alineacion con ODS | Si aplica, conecta con innovacion, procesos o consumo responsable? | La relacion con ODS 9 u ODS 12 es explicita y no decorativa. |

Matriz de decision:

| Alternativa | Problema | Actor | Viabilidad | Aprendizaje | Etica | Total | Decision |
|---|---:|---:|---:|---:|---:|---:|---|
|  |  |  |  |  |  |  | Prototipar / ajustar / descartar |
|  |  |  |  |  |  |  | Prototipar / ajustar / descartar |
|  |  |  |  |  |  |  | Prototipar / ajustar / descartar |

La matriz ayuda a justificar. Sin embargo, la decision final debe explicarse en una nota breve: por que esta alternativa permite aprender mejor que las demas dentro del alcance del curso.

### 6. Prototipo de baja fidelidad

Un prototipo de baja fidelidad es una representacion simple, rapida y economica de una solucion. Su objetivo es provocar conversacion, observacion y retroalimentacion. No busca impresionar por acabado, sino permitir que un actor imagine, use o critique la propuesta.

Brown (2009) plantea que prototipar permite explorar ideas de forma tangible. En emprendimiento inicial, esto es especialmente util porque reduce el riesgo de construir demasiado antes de aprender. Un prototipo de baja fidelidad puede estar hecho en papel, en diapositivas, en una plantilla digital, en un formulario simulado, en un mapa de proceso, en un storyboard o en una maqueta navegable simple.

| Tipo de prototipo | Uso recomendado | Ejemplo |
|---|---|---|
| Boceto en papel | Explorar forma, flujo o disposicion inicial. | Pantallas de una app dibujadas a mano. |
| Storyboard | Mostrar experiencia de uso paso a paso. | Secuencia de como un usuario consulta disponibilidad de equipos. |
| Wireframe | Representar interfaz digital sin programar. | Pantallas basicas de registro de pedidos. |
| Maqueta fisica | Representar objeto, espacio o senalizacion. | Distribucion de puntos de residuos y senales. |
| Prototipo de servicio | Simular interaccion entre actores. | Guion de atencion o rutina de confirmacion de pedidos. |
| Plantilla o formato | Probar estructura de captura o decision. | Registro unico de pedidos o ficha de solicitud. |
| Mapa de proceso futuro | Comparar proceso actual y propuesto. | Flujo antes/despues de reserva de laboratorio. |
| Landing o pieza simulada | Explorar mensaje y propuesta sin vender. | Pagina informativa no publica para recibir retroalimentacion. |

Un prototipo de baja fidelidad debe incluir lo suficiente para que el actor reaccione. Si es demasiado abstracto, no genera aprendizaje. Si es demasiado terminado, el equipo puede resistirse a cambiarlo. El punto medio es una representacion clara, modificable y enfocada en la hipotesis priorizada.

### 7. Alcance de implementacion controlada

La implementacion del Momento 3 no es una implementacion masiva. Es una prueba controlada del prototipo o de su representacion. El equipo debe definir donde, con quien, durante cuanto tiempo y bajo que limites se mostrara o simulara la solucion.

| Elemento | Pregunta guia | Ejemplo de respuesta |
|---|---|---|
| Escenario | En que contexto se presentara o simulara el prototipo? | Entrevista con actor, foro del aula, espacio observado con autorizacion. |
| Participantes | Quienes podran interactuar o retroalimentar? | Usuarios, clientes, decisores o aliados priorizados. |
| Duracion | Cuanto durara la interaccion? | 10 a 20 minutos por actor o una sesion breve de revision. |
| Accion esperada | Que hara el participante? | Interpretar, recorrer, elegir, comentar, simular uso o comparar alternativas. |
| Evidencia | Que se registrara? | Notas anonimas, comentarios, dudas, errores, decisiones y sugerencias. |
| Limites | Que no se hara? | No vender, no recolectar datos sensibles, no publicar imagenes sin permiso. |

Una implementacion controlada debe dejar claro que el prototipo es academico y preliminar. Si el participante cree que interactua con un servicio real, una venta real o un sistema institucional activo, puede generarse confusion o riesgo etico. La transparencia es parte de la calidad academica.

### 9. Guion de uso o recorrido del prototipo

Un prototipo necesita un guion de uso. El guion describe como un actor lo recorrera, que tarea realizara y que observara el equipo. Sin guion, la retroalimentacion puede volverse general y poco util.

| Paso | Accion del participante | Elemento del prototipo | Que observa el equipo | Pregunta de cierre |
|---|---|---|---|---|
| 1 |  |  |  |  |
| 2 |  |  |  |  |
| 3 |  |  |  |  |
| 4 |  |  |  |  |

Ejemplo de guion para microempresa:

| Paso | Accion del participante | Elemento del prototipo | Que observa el equipo | Pregunta de cierre |
|---|---|---|---|---|
| 1 | Recibe un pedido simulado por mensaje. | Plantilla de registro unico. | Si entiende donde registrar datos basicos sin exponer clientes reales. | Que campo le resulta innecesario o falta? |
| 2 | Actualiza estado del pedido. | Flujo de seguimiento. | Si el proceso reduce confusion o agrega carga. | En que momento dejaria de usarlo? |
| 3 | Revisa pedido pendiente. | Vista de control. | Si identifica prioridad y error posible. | Que decision le permite tomar esta vista? |

El guion no debe presionar al actor a decir que la solucion es buena. Debe permitir observar uso, confusion, dudas, rechazo o sugerencias.

### 10. Bitacora de decisiones emprendedoras

La bitacora es una evidencia central del curso porque muestra la evolucion del pensamiento emprendedor. No solo registra que hizo el equipo, sino por que lo hizo y que aprendio. En Gate 1, la bitacora ayuda a demostrar trazabilidad entre problema, evidencia, hipotesis, prototipo y decision.

| Fecha | Decision | Evidencia que la sustenta | Alternativas consideradas | Riesgo identificado | Proximo paso |
|---|---|---|---|---|---|
|  |  |  |  |  |  |
|  |  |  |  |  |  |
|  |  |  |  |  |  |

Buenas entradas de bitacora:

1. Explican una decision concreta.
2. Distinguen evidencia de opinion.
3. Registran cambios, dudas o contradicciones.
4. Reconocen limitaciones de la informacion.
5. Incluyen acciones futuras verificables.

Entradas debiles:

1. "Nos reunimos y avanzamos".
2. "La idea gusto mucho".
3. "Decidimos hacer una app porque es mejor".
4. "Todos estuvieron de acuerdo".
5. "No hubo problemas".

La bitacora tambien debe registrar evidencia negativa. Si el equipo descarta una alternativa, debe explicar por que. Si el prototipo cambia, debe dejar rastro del cambio.

### 11. Preparacion para validacion y evaluacion

El Momento 3 no termina en el prototipo como objeto. Termina con un prototipo listo para ser validado. Por eso, el equipo debe definir que espera aprender en el Momento 4 y como sabra si la solucion debe continuar, pivotar, reformularse o cerrarse.

| Pregunta de preparacion | Respuesta esperada |
|---|---|
| Que hipotesis de solucion se validara? |  |
| Que actor debe interactuar con el prototipo? |  |
| Que tarea realizara ese actor? |  |
| Que senales indicaran comprension o valor? |  |
| Que senales indicaran confusion, rechazo o falta de pertinencia? |  |
| Que datos no se recolectaran por cuidado etico? |  |
| Que decision se tomara segun la evidencia? |  |

Ejemplos de senales:

| Senal | Interpretacion posible | Cuidado |
|---|---|---|
| El actor comprende el flujo sin explicacion excesiva. | El prototipo comunica una logica clara. | No asumir adopcion solo por comprension. |
| El actor identifica una parte util y una innecesaria. | La solucion requiere ajuste. | Registrar ambos comentarios. |
| El actor no reconoce el problema en el prototipo. | Puede haber desalineacion problema-solucion. | Revisar hipotesis y evidencia inicial. |
| El actor pide una funcion que no estaba contemplada. | Puede aparecer nueva necesidad. | No agregar todo sin priorizar. |
| El actor rechaza por tiempo, costo o privacidad. | Hay barrera de adopcion. | Considerarla como evidencia valiosa. |

### 12. Relacion con ODS 9 y ODS 12

La solucion inicial puede conectarse con ODS 9 y ODS 12 si la evidencia del reto lo permite. Esta conexion debe ser concreta. No se trata de decorar el prototipo con un ODS, sino de explicar que parte del problema y de la solucion conversa con innovacion, infraestructura, procesos, produccion responsable o consumo responsable.

| ODS | Conexion con solucion e implementacion | Ejemplo |
|---|---|---|
| ODS 9. Industria, innovacion e infraestructura | Prototipos que mejoran procesos, acceso a informacion, uso de infraestructura o servicios academicos/productivos. | Tablero de disponibilidad de equipos compartidos o flujo de reserva simulado. |
| ODS 12. Produccion y consumo responsables | Prototipos que promueven reduccion de desperdicio, separacion, consumo consciente o gestion responsable de recursos. | Senalizacion de baja fidelidad para separacion de residuos en cafeteria. |

La relacion con ODS debe mantenerse en nivel de oportunidad. El equipo no debe afirmar impacto ambiental, social o economico sin validacion suficiente. Puede decir que el prototipo busca explorar una mejora potencial relacionada con el ODS correspondiente.

### 13. Errores frecuentes en solucion e implementacion

| Error frecuente | Consecuencia | Como corregirlo |
|---|---|---|
| Construir demasiado pronto. | Se invierte tiempo en una solucion no validada. | Usar prototipo de baja fidelidad y enfocar aprendizaje. |
| Elegir la solucion favorita del equipo. | Se pierde trazabilidad con evidencia. | Aplicar matriz evidencia-solucion y criterios de seleccion. |
| Prototipar tecnologia sin necesidad clara. | La tecnologia oculta el problema real. | Considerar formatos fisicos, de servicio o proceso. |
| Presentar el prototipo como producto final. | Genera expectativas falsas o confusion etica. | Aclarar finalidad academica y caracter preliminar. |
| Preguntar si "gusta" la solucion. | Produce retroalimentacion superficial. | Usar tareas, escenarios y preguntas sobre experiencia real. |
| Ignorar restricciones del contexto. | La solucion no podria implementarse responsablemente. | Revisar permisos, roles, recursos y privacidad. |
| No documentar cambios. | Se pierde trazabilidad para Gate 1. | Mantener bitacora con evidencia y decisiones. |
| Ocultar evidencia negativa. | Debilita rigor academico. | Registrar contradicciones y ajustar la solucion. |

### 14. Cierre conceptual del Momento 3

Al finalizar este momento, el estudiante debe tener una solucion inicial representada, no una promesa abstracta. El prototipo debe mostrar que el equipo entiende el problema, reconoce al actor, traduce evidencia en criterios y prepara una validacion etica.

El Momento 3 queda completo cuando el equipo puede responder:

1. Que evidencia justifica la solucion inicial?
2. Que hipotesis prioritaria se conecta con el prototipo?
3. Que propuesta de valor inicial se esta explorando?
4. Que tipo de prototipo de baja fidelidad se construyo y por que?
5. Que elementos se incluyeron y cuales se dejaron por fuera?
6. Como se implementara de manera controlada?
7. Que decisiones quedaron registradas en la bitacora?
8. Que se validara en el Momento 4?

Una solucion inicial fuerte no es la mas completa, sino la que permite aprender con menor costo, menor riesgo y mayor trazabilidad. En emprendimiento temprano, construir bien significa construir lo suficiente para escuchar mejor.

| Arquitectura, diseno, creatividad, territorio o sostenibilidad | Usuarios dudan sobre separacion de residuos por senalizacion poco clara o ubicacion de puntos. | Sistema visual de senalizacion y recorrido de decision para disposicion de residuos. | Bocetos de senales, maqueta de ubicacion, tarjetas con ejemplos de residuos y storyboard de uso. | Mostrar piezas a usuarios o pares; pedir que clasifiquen ejemplos ficticios y observar dudas. | No fotografiar rostros ni culpabilizar usuarios; evitar afirmaciones ambientales no validadas. |

## Relacion con evaluacion

| Evidencia del Momento 3 | Criterios de evaluacion asociados | Uso en el portafolio Gate 1 |
|---|---|---|
| Matriz evidencia-solucion | Coherencia problema-hipotesis-evidencia-solucion. | Demuestra que la solucion surge del aprendizaje y no solo de una ocurrencia. |
| Propuesta de valor inicial | Claridad del actor, dolor, ganancia esperada y prudencia academica. | Explica el valor potencial de la solucion de forma verificable. |
| Prototipo de baja fidelidad | Pertinencia, claridad, alcance, modificabilidad y conexion con hipotesis. | Permite mostrar la solucion inicial y preparar validacion. |
| Ficha tecnica del prototipo | Trazabilidad, criterios de diseno, limites y riesgos. | Documenta decisiones y alcance de implementacion controlada. |
| Bitacora de decisiones | Reflexion, evidencia usada, cambios, alternativas y proximo paso. | Sustenta la narrativa de aprendizaje para la sustentacion Gate 1. |
| Guion de implementacion controlada | Viabilidad, etica y claridad de la interaccion con actores. | Prepara la recoleccion de evidencia en el Momento 4. |

La rubrica Gate 1 evaluara este momento principalmente en los criterios de prototipo de baja fidelidad, coherencia de la solucion con el problema, evidencia temprana, trazabilidad de decisiones, comunicacion, etica y preparacion para validacion. Un buen desempeno no depende de entregar una solucion sofisticada, sino de construir una representacion clara, justificable y abierta a cambio.

## Criterios eticos/legales

| Criterio | Orientacion para el estudiante |
|---|---|
| Transparencia | Indique que el prototipo es academico, preliminar y no corresponde a una oferta real. |
| Consentimiento | Solicite autorizacion antes de registrar comentarios, imagenes, audio o datos identificables. |
| Minimizacion de datos | No pida informacion personal, comercial o institucional que no sea necesaria para aprender. |
| Confidencialidad | Use datos ficticios o anonimizados en prototipos de empresas, procesos o clientes. |
| Propiedad intelectual | No use marcas, logos, imagenes, software, planos o materiales de terceros sin permiso o fuente valida. |
| Accesibilidad | Procure que el prototipo sea legible, comprensible y no excluya innecesariamente a usuarios. |
| No prometer impacto | No declare beneficios ambientales, sociales, economicos o tecnicos que no se hayan validado. |
| No simular servicios reales | Evite hacer creer que una plataforma, venta, campana o sistema institucional esta activo si no lo esta. |
| Seguridad | No prototipe intervenciones que puedan poner en riesgo personas, equipos, informacion o instalaciones. |
| Uso responsable de IA | Si usa IA para bocetar, redactar o sintetizar, revise sesgos, no suba datos sensibles y documente apoyo cuando aplique. |
